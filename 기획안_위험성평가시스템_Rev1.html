문제 / 수치화

현재 위험성평가는 엑셀과 수기 문서로 관리된다. 복수 사업장·다수 협력사의 평가 이력, 결재, 개선조치 이행을 일관되게 추적할 수 없고, 법정 기한 누락과 증빙 산출 지연의 리스크가 상시 존재한다.

핵심 문제 4가지

  • 이력 추적 불가 — 사업장 20곳, 협력사 200개사의 평가서가 개별 엑셀 파일로 흩어져 있어 누가·언제·무엇을 평가하고 조치했는지 통합 조회가 불가능하다.
  • 법정 기한 누락 위험 — 정기평가(매 1년), 수시평가(사유 발생 시 작업 개시 전), 최초평가(성립 후 1개월 내 착수)의 도래·초과를 사람이 수동 관리한다.
  • 개선조치 이행 블랙박스 — 감소대책의 담당자·기한·증빙·재평가가 문서에만 남아 이행 여부를 확인하려면 담당자에게 일일이 물어야 한다.
  • 증빙 산출 지연 — 중대재해처벌법·감독 대응 시 평가서 + 결재 이력 + 참여·서명 기록 + 조치 이력을 취합하는 데 며칠이 걸린다. 위험성평가 기록은 중처법 시행령 §4 대응의 핵심 증빙이다.

개발 목표

HAZOP·KRAS·JSA·CHARM 4개 기법의 위험성평가 전 과정을 하나의 웹앱으로 일원화한다. 핵심 설계 원칙은 “공통 평가 엔진 + 기법별 템플릿” — 결재·이행관리·공유·기록·알림·일정은 공통 워크플로 하나를 쓰고, 기법별로는 시트(입력 폼)와 위험도 산정 로직만 달리한다.

G5근로자 참여·공유 의무 이행열람 서명률 대상자의 95% 이상

단계별 릴리스 계획 (§10)

Phase범위완료 기준
Phase 1 — 코어 KRAS·JSA 작성 + 전자결재 + 조치 이행 + 공유·서명 + 3년 기록. 입력 가이드라인, 고위험 자동알림, 수급업체 이행점검 체크리스트, JSA 현장확인 포함 파일럿 1개 사업장에서 정기(KRAS)·협력사(JSA) 평가의 작성→결재→공유·서명→조치 완료 사이클이 시스템만으로 완결되고 증빙 패키지가 출력된다
Phase 2 — 확장 HAZOP·CHARM 기법, 상시평가 모드(월·주 사이클), 위험요인 제보, 연간 실시계획, 표준모델, 도급 안전보건정보 제공 파일럿 사업장의 정기+수시 운영 안정 후 상시평가 모드 전환
Phase 3 — 고도화 통계·리포트, 요인 라이브러리 고도화, QR 발행, 본사 계획 전개, 산보위 보고, 대결(대리결재) 등 운영 편의 —
기술 스택은 PRD에서 의도적으로 미지정(설계 단계 결정 사항)이다. 본 기획안 §6은 제안이며, 세부 실행 PRD 수립 시 확정한다.

협업 방법

이 프로젝트의 협업은 세 층위로 나뉜다 — ① 문서(PRD) 중심의 요구사항 협업, ② 개발팀 내부 협업, ③ 현업(안전부서·법무·현장)과의 확인 협업.

① 문서 중심 협업 — PRD가 단일 진실 원천(SSOT)

  • PRD_origin_Rev1(v2.2)을 요구사항의 원본으로 삼고, 모든 기능은 FR-ID(예: FR-ASMT-003)로 추적한다. 구현 이슈·테스트 케이스에 FR-ID를 연결해 요구사항 커버리지를 관리한다.
  • 절차서 우선 원칙: 회사 절차서(KDHC-SHP-04 R7)와 PRD가 충돌하면 절차서 기준을 채택하고, 절차서에 없는 시스템 요구(검증 규칙·화면·데이터 관리)는 PRD를 유지한다(§5.2.3).
  • 결정 기록 관리: 기준 차이 항목은 별도 문서(PRD-지침서_기준차이_확인필요사항.md)로 정리해 일자 기입 결정(2026-07-08, 9개 항목)으로 종결하고, 개정분은 문서 내 색상 표시(🔵)로 리뷰 가능하게 남긴다.
  • 오픈 이슈 보드: 미결 사항은 PRD §13 오픈 이슈 표로 관리 — 담당·다음 조치를 명시하고 해결 시 취소선+해결 버전을 기록한다.

② 개발팀 협업

  • 버전 관리: Git + GitHub. 브랜치 전략은 GitHub Flow(main 보호 + feature 브랜치 + PR)
  • 코드 리뷰: 모든 PR은 리뷰 승인 후 병합. PR 설명에 대응 FR-ID·화면 ID(S-xx) 명시
  • 이슈 트래킹: GitHub Issues/Projects — Phase 1~3 마일스톤과 매핑
  • 화면 설계: Figma로 S-01~S-48 와이어프레임 → 디자인 확정 후 구현
  • 진행 공유: 데일리 스탠드업(15분) + 주간 스프린트 리뷰
  • 수용 기준 테스트: URL 직접 접근 차단 등 권한 테스트를 수용 기준에 포함(NFR-006)

③ 현업 확인 협업

  • 안전부서: 상시평가 ‘일 단위 주지’ 갈음 요건 해석(이슈 #4), 기법 범위 편입(이슈 #11)
  • 법무: 전자서명 증빙력 수준 — 터치 서명 vs 본인인증 결합(이슈 #2), 협력사 근로자 개인정보 동의 절차(이슈 #3)
  • 현장 운영: JSA 현장확인 서명 주체 방식(이슈 #10), 평가서 번호 체계·종이 문서 소급 등록(이슈 #5)
  • 공정안전 담당: HAZOP 회의록·PSM 문서 세트 연동 범위 인터뷰(이슈 #6) → Phase 2 설계 반영
제품 자체도 협업 도구다 — 전자결재(작성→검토→승인), 협력사 제출·반려·재제출 플로우, 공유 발행·열람 서명이 시스템 안에서 도급인·수급인 간 협업 기록으로 남아 법정 증빙이 된다.

환경

사용자 실행 환경과 시스템 운영 환경의 기준은 비기능 요구사항(NFR-001~011)으로 확정되어 있다.

사용자 환경

  • 플랫폼: 웹 + 모바일 반응형 단일 웹앱 — 네이티브 앱 없음
  • 브라우저: 최신 Chrome / Edge / Safari + 모바일 Safari / Chrome. IE 미지원 (NFR-010)
  • 모바일: 근로자 대상 화면(공유·서명, 제보, 내 조치, 현장확인)은 360px 폭 완전 동작 (NFR-004)
  • 네트워크: 현장 저속(3G 수준)에서 공유 화면 5초 이내 로딩 — 이미지 지연 로딩
  • 접근성: 큰 글씨·3탭 이내 서명 완료 동선, 위험등급 색상에 텍스트 라벨 병기(색약 대응, NFR-011)
  • 언어: 한국어 (다국어는 향후 과제)

운영·보안 환경

  • 성능: 일반 화면 P95 3초 이내, 동시 접속 300명, 대시보드 10초 이내 또는 사전 집계 (NFR-002)
  • 가용성: 업무시간(06:00–20:00) 가용률 99.5%, 계획 점검은 야간 공지 후 (NFR-003)
  • 백업·복구: 일 1회 이상 백업, RPO 24시간 / RTO 8시간 (NFR-008)
  • 데이터 보존: 평가·결재·서명·조치·감사 기록 3년 이상 불변 보존 — 보존 기한 내 삭제 기능 자체를 제공하지 않음 (FR-REC-002)
  • 보안: 전 API 서버 측 스코프·역할 검증, 협력사 간 데이터 격리 필수, URL 직접 접근 차단 테스트 (NFR-006)
  • 개인정보: 수집 최소화(이름·소속·연락처·서명), 법정 보존 기록은 파기보다 보존 우선 (NFR-007)
  • 배포 환경(제안): 개발 → 스테이징 → 운영 3단계 분리, 운영 반영은 야간 배포
