## 1. 개요

### 1.1 배경

- 산업안전보건법 제36조는 모든 사업주에게 유해·위험요인을 파악하고 위험성을 평가·개선할 의무를 부과하며, 결과와 조치사항의 **3년 기록·보존**(시행규칙 제37조)을 요구한다.
- 2023년 고시 개정(제2023-19호)으로 **상시평가 제도**(월·주·일 주기)가 신설되었고, 2024년 개정(제2024-76호)으로 참여·공유 대상이 **고용형태·국적 불문 사업장 내 모든 근로자**(협력사 포함)로 명확화되었다.
- 중대재해처벌법 시행령 제4조 제3호에 따라 위험성평가 절차를 마련하여 실시하고 그 결과를 경영책임자가 보고받으면 반기 1회 유해·위험요인 확인·개선 점검 의무를 이행한 것으로 간주된다. → 위험성평가 기록은 중처법 대응의 **핵심 증빙**이다.
- 현재 엑셀·수기 중심의 평가 관리로는 복수 사업장·다수 협력사의 평가 이력, 결재, 개선조치 이행을 일관되게 추적하기 어렵고, 법정 기한(정기평가 연 1회, 수시평가 트리거) 누락 위험이 있다.

### 1.2 제품 목적

건설업·제조업 복수 사업장을 운영하는 기업이 **자사 근로자와 협력사(수급업체) 근로자가 함께 사용하는 개방형 시스템**에서:

1. HAZOP·KRAS·JSA·CHARM 4개 기법의 위험성평가를 작성 → 결재 → 공유 → 기록까지 온라인으로 일원화하고,
2. 감소대책(개선조치)의 이행을 담당자·기한·증빙·재평가까지 추적하며,
3. 최초·정기·수시·상시평가의 법정 실시 주기를 자동 관리하고,
4. 본사가 전 사업장·협력사의 이행 현황을 통합 대시보드로 관제한다.

### 1.3 제품 구조 원칙 (승인된 설계 방향)

**공통 평가 엔진 + 기법별 템플릿** 구조를 채택한다.

- 결재, 이행관리, 공유·서명, 기록·보존, 알림, 일정 관리는 4개 기법이 **하나의 공통 워크플로**를 공유한다.
- 기법별로 다른 것은 **평가 시트(입력 폼)의 필드 구성과 위험도 산정 로직**뿐이다.
- HAZOP처럼 구조가 특수한 기법은 시트 레이아웃만 전용으로 설계한다.
- 위험도 매트릭스·허용기준은 **회사 표준(전 기법 공통 4×4, 5단계 등급 — §5.2)**을 디폴트로 탑재하되, 관리자가 수정할 수 있는 **설정형**으로 구현한다.

### 1.4 범위 요약

| 구분 | 포함 | 제외 |
|---|---|---|

### 2.1 제품 목표

| # | 목표 | 측정 방법 |

- 수시평가 트리거 이벤트 등록 → 평가 완료 리드타임 평균 7일 이내
- 협력사 관리자 계정 활성률 80% 이상
- 월간 활성 사용자(MAU) 전체 등록 사용자의 60% 이상

## 4. 정보 구조 (조직·데이터 스코프)

### 4.1 조직 계층

```
회사(자사)
 ├─ 본사 (통합 관제)
 ├─ 사업장/현장 #1 (제조 공장 또는 건설 현장)
 │   ├─ 부서/공정 #1-1
 │   │   └─ 작업(Job) / 설비 / 화학물질
 │   └─ 부서/공정 #1-2
 ├─ 사업장/현장 #2 ...
 └─ 협력사(수급업체) A, B, C ...
     └─ 협력사–사업장 계약 관계 (어느 사업장에서 어떤 작업을 수행하는지)
```

- **사업장/현장**: 평가·결재·일정 관리의 기본 단위. 제조업 공장과 건설업 현장을 동일 개념으로 다루되, 유형 속성(제조/건설)을 가진다. 건설 현장은 개설일·준공예정일(현장 수명주기)을 가진다.
- **협력사**: 회사 레벨에 마스터로 등록하고, 사업장별 **계약(투입) 관계**로 연결한다. 하나의 협력사가 여러 사업장에 투입될 수 있다.
- **작업(Job)/설비/화학물질**: 평가 대상 마스터. JSA는 작업, HAZOP은 공정/설비(노드), CHARM은 화학물질을 평가 대상으로 참조한다.

### 4.2 평가서의 스코프 및 용도 (회사 표준)

| 기법 | 용도 (회사 규정) | 평가 대상 단위 | 주 작성 주체 | 실시 유형 연계 |
|---|---|---|---|---|
| **HAZOP** | 산업안전보건법상 **공정안전자료(PSM)**에 포함되는 공정위험성평가 | 사업장 > 공정 > 검토구간(Node) | 자사 공정담당 + 평가팀(스터디) | 최초 / 정기(공정안전보고서 주기 갱신) |
| **KRAS** | **정기 사업장 위험성평가** — 부서·공정·작업 전반을 주기적으로 재점검 | 사업장 > 부서/공정 > 작업·설비 | 자사 부서담당 | 정기(연 1회) / 상시(월 사이클) |
| **JSA** | **신규 작업 발생 시** 작업 착수 전 실시하는 작업위험성평가 | 사업장 > 작업(Job) 단위 | 자사 부서담당 / **협력사 관리자** | **수시**(신규 작업 도입 트리거와 자동 연계 — §5.6 FR-SCHD-003) |
| **CHARM** | 사업장이 **저장·취급하는 화학물질**에 대한 물질위험성평가 | 사업장 > 부서 > 취급 화학물질 | 자사 보건담당 | 정기 / 수시(신규 물질 도입 시) |

> 4개 기법은 용도(공정/정기/신규작업/화학물질)가 서로 달라 병행 운영되며, 하나가 다른 것을 대체하지 않는다. 다만 위험성 결정에는 **공통 4×4 매트릭스**(§5.2)를 사용해 기법 간 위험도 수치를 조직 전체에서 동일한 기준으로 비교할 수 있게 한다.

## 5. 기능 요구사항 (Functional Requirements)

표기 규칙: `FR-<모듈>-<번호>`, 우선순위 P0(필수·법정) / P1(핵심) / P2(고도화).
각 요구사항의 릴리스 단계는 §10 릴리스 계획의 Phase 표기를 따른다.

### 5.1 조직·계정 관리 (ORG)

| ID | 우선순위 | 요구사항 |
|---|---|---|
| FR-ORG-001 | P0 | 시스템 관리자는 사업장/현장을 등록·수정·비활성화할 수 있다. 속성: 명칭, 유형(제조/건설), 주소, 상시근로자 수, 개설일, (건설) 준공예정일, 안전관리자 지정. |
| FR-ORG-002 | P0 | 사업장 하위에 부서/공정을 트리 구조로 등록할 수 있다(최대 3단계). |

### 5.2 실시규정·위험도 기준 설정 (CFG)

> 고시 제9조(사전준비): 위험성평가 실시규정, 위험성 수준과 판단기준, 허용 가능한 위험성 기준을 **평가 실시 전에** 확정해야 한다. 시스템은 이를 설정 데이터로 관리한다.

| ID | 우선순위 | 요구사항 |
|---|---|---|
| FR-CFG-001 | P0 | 회사/사업장 단위로 **위험성평가 실시규정 문서**를 등록·게시할 수 있다(파일 첨부 + 시행일 + 개정 이력). 실시규정에는 평가 방법·절차·시기, 담당자·책임자 역할, 근로자 참여·공유 방법, 기록·보존 방법이 포함된다. |
| FR-CFG-002 | P0 | **회사 표준 위험도 매트릭스**(4개 기법 공통)를 시스템 디폴트로 제공한다: 두 축(발생빈도/가능성 × 발생강도/중대성) 각 1–4단계, **4×4 = 16점 척도**. 산정식은 `위험성 = 축1 × 축2`. 등급·조치기한 구간은 아래 §5.2.1 표를 디폴트로 하며, 관리자가 축 단계 수·정의문·구간·색상·기준선을 수정할 수 있다(기법별로 축 정의(예: HAZOP은 빈도×강도, CHARM은 유해성×노출수준)만 달리하고 척도·등급 구간은 공통 유지가 원칙). |
| FR-CFG-003 | P0 | 매트릭스·기준 변경은 **버전 관리**된다. 기존 평가서는 작성 당시 버전의 기준으로 표시·보존되고, 변경은 신규 평가서부터 적용된다. |
| FR-CFG-004 | P0 | 위험 등급별 **후속 조치 규칙**(§5.2.1 표 기준)을 설정한다: 등급별 조치 착수 기한(즉시/조속/1개월 내/계획 수립)과 승인권자 상향 여부를 등급 구간에 매핑한다. 위험성 **5점 미만(1~4)은 개선조치 수립·시행 불필요**로 자동 판정한다. |
| FR-CFG-005 | P1 | 유해·위험요인 분류 체계(기계적/전기적/화학적/생물학적/작업특성/작업환경 요인 등 KOSHA 분류)를 코드 마스터로 제공하고 관리자가 항목을 추가할 수 있다. |
| FR-CFG-006 | P1 | 결재선 템플릿 설정: 사업장·기법별로 검토자/승인자 디폴트 지정(§5.4 참조). |
| FR-CFG-007 | P2 | 안전보건정보 자료실: 재해사례, MSDS, 설비 사양서, 작업환경측정 결과 등 사전조사 자료를 등록하고 평가서에서 참조·첨부할 수 있다(고시 제9조 사전조사 + 시행규칙 제37조 기록사항). |

#### 5.2.1 회사 표준 위험성 결정 매트릭스 (전 기법 공통)

**4×4 매트릭스** (가능성 1~4 × 중대성 1~4, 위험성 = 두 축의 곱, 범위 1~16)

| 위험성 점수 | 등급 | 조치 요구사항 |
|---|---|---|
| **13~16** | 매우 위험 | **즉시** 개선조치 수립 및 시행 |
| **10~12** | 위험 | **빠른 시일 내에** 개선조치 수립 및 시행 |
| **7~9** | 다소 위험 | **1개월 내에** 개선조치 수립 및 시행 |
| **5~6** | 위험성 존재 | 개선조치 수립 및 시행 **필요** |
| **1~4** | 허용 가능한 위험 | 개선조치 수립 및 시행 **불필요** |

- **판정 기준선**: 위험성 **5점 이상**인 모든 요인은 감소대책(개선조치)이 **필수**이며, 미입력 시 결재 상신이 차단된다(FR-ASMT-003 연동).
- 이 매트릭스는 KRAS·JSA·HAZOP·CHARM **4개 기법에 공통 적용**되는 회사 표준이다. 기법별로 두 축의 의미만 다르다: KRAS·JSA·HAZOP은 (발생빈도×발생강도), CHARM은 (유해성 등급×노출수준 등급). 축의 명칭·정의문은 §5.4의 기법별 시트에서 정의한다.
- 관리자는 등급 구간·조치기한·색상을 FR-CFG-002/004에 따라 조정할 수 있으나, 변경 시 버전 관리(FR-CFG-003)가 적용된다.

**공통 기능 요구사항**

| ID | 우선순위 | 요구사항 |
|---|---|---|
| FR-ASMT-001 | P0 | 사용자는 기법을 선택해 평가서를 생성하고, 임시저장(Draft) 상태로 여러 차례 나누어 작성할 수 있다. |
| FR-ASMT-002 | P0 | 평가서의 각 행(요인 항목)은 공통적으로 다음을 가진다: ① 유해·위험요인(분류코드 + 서술) ② 관련 재해 형태(떨어짐·끼임·부딪힘 등 KOSHA 재해분류) ③ 현재 안전보건 조치 ④ 현재 위험성(기법별 축 값 × 회사 표준 4×4 매트릭스 산정, §5.2.1) ⑤ 개선조치 필요 여부(5점 이상 자동 판정) ⑥ 감소대책(5점 이상 시 필수) ⑦ 개선 후 예상 위험성 ⑧ 조치 담당자·기한. |
| FR-ASMT-003 | P0 | 위험성은 축 값 선택 시 **자동 계산**(축1×축2)되고 §5.2.1 등급 색상으로 표시된다. **위험성 5점 이상**인 항목에 감소대책이 없으면 결재 상신이 차단된다. |
| FR-ASMT-004 | P0 | 감소대책 입력 시 **대책 유형 분류를 필수**로 선택한다: ① 제거·대체(본질적 대책) ② 공학적 대책 ③ 관리적 대책 ④ 개인보호구. UI는 법정 우선순위 순서로 노출하고, ③·④만 선택된 고위험 항목에는 상위 대책 검토를 권고하는 경고를 표시한다(고시 제12조). |
| FR-ASMT-005 | P0 | 평가팀·참여 근로자는 사용자 검색으로 추가한다. 참여 근로자에는 협력사 근로자도 포함할 수 있다. 참여자별 역할(리더/평가자/참여근로자)이 기록된다. |
| FR-ASMT-006 | P1 | **요인 라이브러리**: 승인 완료된 과거 평가서의 요인·대책 행을 검색해 복사할 수 있다. 작업/공정별 자주 쓰는 요인 세트를 템플릿으로 저장·재사용할 수 있다. |
| FR-ASMT-007 | P1 | 정기평가 생성 시 직전 평가서를 **복제 후 갱신** 모드로 열 수 있다(고시 제15조: 정기평가는 기존 결과의 적정성 재검토). 변경된 행에는 변경 표시가 남는다. |
| FR-ASMT-008 | P1 | 행 단위 사진·파일 첨부(현장 사진, 도면 발췌). 모바일 카메라 촬영 직접 업로드 지원. |
| FR-ASMT-009 | P1 | 평가서 출력: 인쇄용 양식(PDF)과 엑셀 다운로드. 출력물에는 결재 이력·참여자·서명 현황이 포함된다. |
| FR-ASMT-010 | P2 | 평가서 개정(Revision): 승인 완료된 평가서를 개정하면 새 버전이 생성되고 이전 버전은 읽기 전용으로 보존된다. |

**평가서 상태 머신 (공통)**

```
작성중(Draft) → 결재중(검토→승인) → 승인완료(Approved)
                     ↘ 반려(Rejected) → 작성중으로 복귀
승인완료 → 공유중(근로자 열람·서명 진행) → 조치진행(개선조치 이행) → 완료(Closed)
승인완료 이후 → 개정(Revision) 시 새 버전 Draft 생성
```

- 승인완료 이후의 평가서 본문은 **수정 불가**(개정으로만 변경). 조치 이행 상태와 서명 현황만 갱신된다.

### 5.4 기법별 평가 시트 (ASMT-H/K/J/C)

#### 5.4.1 KRAS — 빈도·강도법 정기 위험성평가 (ASMT-K)

**용도**: 정기 사업장 위험성평가(연 1회, 상시평가 모드 시 월 사이클). 부서/공정의 작업·설비 전반을 대상으로 한다.

**행 필드 구성** (§5.2.1 회사 표준 4×4 매트릭스 적용)

| 열 | 필드 | 입력 방식 |
|---|---|---|
| 1 | 공정/작업명 | 작업 마스터 선택 또는 직접 입력 |
| 2 | 세부 작업내용 | 텍스트 |
| 3 | 유해·위험요인 분류 | 코드 선택(기계적/전기적/화학적/생물학적/작업특성/작업환경) |
| 4 | 유해·위험요인 및 발생상황 | 텍스트 (원인과 결과가 드러나게 서술) |
| 5 | 관련 법규·근거 | 선택 입력 |
| 6 | 현재 안전보건조치 | 텍스트 |
| 7 | 발생빈도(가능성) | 1–4 선택 |
| 8 | 발생강도(중대성) | 1–4 선택 |
| 9 | 위험성 | 자동: 빈도×강도 (1–16), §5.2.1 5단계 등급·색상 표시 |
| 10 | 개선조치 필요 여부 | 자동 판정 (5점 이상 = 필요, 1~4 = 불필요) |
| 11 | 감소대책 (유형분류 필수) | 5점 이상 시 필수 |
| 12 | 개선 후 위험성 | 빈도·강도 재선택 → 자동 계산 |
| 13 | 조치 담당자/기한 | 사용자·날짜 선택 (§5.2.1 등급별 기한 디폴트 자동 제안) |

- FR-ASMT-K-001 (P0): 위 시트를 표 형태로 다건 입력·행 복사·정렬할 수 있다.
- FR-ASMT-K-002 (P1): 부서/공정 선택 시 해당 조직의 작업 마스터 목록을 불러와 행을 일괄 생성할 수 있다.

#### 5.4.2 JSA — 작업안전분석 (ASMT-J)

**용도**: **신규 작업 발생 시** 작업 착수 전 실시하는 수시평가. 작업(Job) 하나를 단계(Step)로 분해하여 평가한다. **협력사 관리자가 작성·제출하는 기본 기법**이다.

**구조**: 평가서(작업 1건) > 작업단계(Step, 순번) > 단계별 요인 행(복수)

**작업단계 필드**: 순번, 단계명(동사형 서술), 단계 설명, 사용 장비·공구

**단계별 요인 행 필드** (§5.2.1 회사 표준 4×4 매트릭스 적용)

| 열 | 필드 | 입력 방식 |
|---|---|---|
| 1 | 유해·위험요인 | 분류코드 + 서술 |
| 2 | 재해 형태 | KOSHA 재해분류 선택 |
| 3 | 현재 안전조치 | 텍스트 |
| 4 | 발생빈도 | 1–4 선택 |
| 5 | 발생강도 | 1–4 선택 |
| 6 | 위험성 | 자동: 빈도×강도 (1–16), §5.2.1 5단계 등급 표시 |
| 7 | 감소대책(유형분류, 5점 이상 필수) / 개선 후 위험성 / 담당·기한 | 공통 규칙 동일 |

**JSA 헤더 추가 필드**: 작업 위치, 작업 예정일시, 필요 자격·면허, 필요 보호구 체크리스트, 관련 작업허가 여부(참조 텍스트).

- FR-ASMT-J-001 (P0): 단계 추가·삭제·순서 변경(드래그)이 가능하다.
- FR-ASMT-J-002 (P0): 협력사 관리자가 작성하는 JSA는 §5.5 협력사 제출 플로우를 따른다.
- FR-ASMT-J-003 (P0): JSA는 **신규 작업(수시평가 트리거 사유 ④ 작업방법·절차 신규 도입)과 자동 연계**된다. 수시평가 트리거 이벤트(§5.6 FR-SCHD-003)에서 '신규 작업'을 선택하면 JSA 평가서 생성이 강제되고, 작업 개시 예정일까지 승인이 완료되지 않으면 경보가 발생한다(FR-SCHD-004).
- FR-ASMT-J-004 (P1): 유사 작업의 승인된 JSA를 템플릿으로 복제해 새 작업에 적용할 수 있다.

**S-13 JSA 시트 — 화면 설계 상세**

화면은 3단 구성이다.

1. **상단 고정 영역(요약바)**: 평가서 번호, 작업명, 실시유형(수시) 배지, 상태(작성중/결재중/승인완료) 배지, 저장 버튼(자동저장 + 수동저장), "결재 상신" 버튼(위험성 5점 이상 항목에 감소대책이 비어 있으면 비활성화되고 사유 툴팁 표시).
2. **좌측 패널 — 작업단계 트리**: 세로 스텝 리스트(①②③…). 각 스텝 카드는 단계명 요약과 해당 스텝 내 최고 위험등급 색상 배지를 보여준다(요인 미입력 스텝은 회색 "미평가" 배지). 드래그 핸들로 순서 변경, 하단 "+ 단계 추가" 버튼. 스텝을 클릭하면 우측 패널이 해당 스텝으로 전환된다.
3. **우측(메인) 패널 — 선택된 단계 상세**
   - 상단: 단계명(동사형 서술 placeholder 안내, 예: "비계를 설치한다"), 단계 설명, 사용 장비·공구(태그 입력).
   - 하단: 유해·위험요인 행 테이블(열 1~7). 행 추가 시 인라인 아코디언으로 확장. 빈도×강도는 숫자 드롭다운 대신 **4×4 그리드 팝오버**(가로=빈도, 세로=강도, 셀 클릭으로 즉시 값 반영)로 선택하며, 계산된 위험성 점수는 §5.2.1 색상 배지로 즉시 표시된다.
   - 위험성 5점 이상 행은 감소대책 입력란이 붉은 테두리 + 필수(*) 표시로 강조된다. 유형(제거·대체/공학적/관리적/개인보호구) 미선택 상태로 임시저장은 가능하나 결재 상신은 차단된다.
4. **헤더 폼 진입점**: 상단 "작업 정보" 탭에서 별도 패널로 열리며, 공통 헤더(§5.3) 아래 JSA 전용 필드(작업 위치, 작업 예정일시, 필요 자격·면허, 필요 보호구 체크리스트, 관련 작업허가 여부)를 이어서 배치한다.
5. **모바일(협력사 관리자 P5) 대응**: 좌측 트리 + 우측 상세의 2단 레이아웃 대신, 스텝을 아코디언 카드로 세로 나열하는 1단 레이아웃으로 전환한다. 빈도×강도 선택은 풀스크린 4×4 그리드 모달로 표시하고, 요인 행 하단 카메라 아이콘으로 즉시 촬영·업로드한다(FR-ASMT-008).
6. **트리거 연계 배너**: 수시평가 트리거(FR-ASMT-J-003)로 생성된 JSA는 화면 상단에 "신규 작업 트리거: [트리거명] · 작업 개시 예정일 D-n" 배너를 표시하고, 클릭 시 트리거 등록 화면(S-21)으로 이동한다. D-day 이내 미승인 시 배너가 경고색으로 전환된다.

#### 5.4.3 HAZOP — 공정 위험성평가 (ASMT-H)

**용도**: 산업안전보건법상 **공정안전자료(PSM, Process Safety Management)**에 포함되는 공정위험성평가. 연속 공정(화학·유틸리티 설비 등)을 검토구간(Node)과 가이드워드로 체계 분석한다. 팀 검토 회의 기반.

**구조**: 평가서(공정/스터디 1건) > 검토구간(Node, 복수) > 이탈(Deviation) 행(복수)

**평가서 헤더 추가 필드**: 대상 공정명, 참조 도면(P&ID 번호·파일 첨부), 검토 회의 일자(복수 세션), 스터디 리더·서기 지정, **공정안전보고서(PSM) 문서번호**(해당 시 연결)

**검토구간(Node) 필드**: 노드 번호, 구간 설명(시작점~끝점), 설계 의도(Design Intent: 유량·압력·온도 등 정상 운전 조건)

**이탈 행 필드** (§5.2.1 회사 표준 4×4 매트릭스 적용)

| 열 | 필드 | 입력 방식 |
|---|---|---|
| 1 | 공정 변수 | 선택: 유량, 압력, 온도, 액위, 조성, 반응, 시간, 기타 |
| 2 | 가이드워드 | 선택: No/Not, More, Less, Reverse, Part of, As well as, Other than |
| 3 | 이탈(Deviation) | 자동 조합 + 서술 보완 (예: "유량 없음 — No Flow") |
| 4 | 원인(Cause) | 텍스트, 복수 등록 |
| 5 | 결과(Consequence) | 텍스트 |
| 6 | 현재 안전장치(Safeguard) | 텍스트 (인터록, 알람, PSV 등) |
| 7 | 발생빈도 | 1–4 선택 |
| 8 | 발생강도 | 1–4 선택 |
| 9 | 위험성 | 자동: 빈도×강도 (1–16), §5.2.1 5단계 등급 판정 |
| 10 | 개선권고사항(Recommendation) | 5점 이상 시 필수, 대책 유형분류 |
| 11 | 조치 담당자/기한 | 공통 규칙 동일 |

- FR-ASMT-H-001 (P0): 노드·이탈 행의 트리형 입력과 노드별 진행률 표시.
- FR-ASMT-H-002 (P0): 평가서를 **공정안전자료(PSM) 문서 세트의 첨부물**로 출력할 수 있는 전용 양식(FR-ASMT-009 확장)을 제공한다.
- FR-ASMT-H-003 (P1): 가이드워드×공정변수 조합 매트릭스에서 검토 완료 여부를 체크해 누락 조합을 확인할 수 있다.
- FR-ASMT-H-004 (P2): 개선권고사항(Recommendation) 목록만 별도 시트로 추출·출력(HAZOP Close-out 관리).

#### 5.4.4 CHARM — 화학물질 위험성평가 (ASMT-C)

**용도**: 사업장이 **저장·취급하는 화학물질**에 대한 물질위험성평가. 취급 화학물질별로 유해성 등급 × 노출수준 등급으로 위험성을 산정한다(KOSHA CHARM 기법 기반).

**구조**: 평가서(부서/공정 1건) > 화학물질 행(복수)

**화학물질 행 필드** (§5.2.1 회사 표준 4×4 매트릭스 적용 — 축은 유해성×노출수준)

| 열 | 필드 | 입력 방식 |
|---|---|---|
| 1 | 화학물질명 / CAS No. | 화학물질 마스터 선택(MSDS 첨부 연동) |
| 2 | 취급 공정·용도 | 텍스트 |
| 3 | 유해성 등급 (1–4) | 두 가지 산정 경로 지원: ① 노출기준(TWA) 기반 자동 제안 ② GHS 분류(H-code, CMR 여부) 기반 자동 제안. CMR(발암성 1A/1B 등) 물질은 자동으로 최고 등급(4). |
| 4 | 노출수준 등급 (1–4) | 두 가지 산정 경로 지원: ① 작업환경측정 결과 입력 시 측정치/노출기준 비율로 자동 산정 ② 측정 결과 없으면 하루 취급량(대/중/소) × 비산성·휘발성(고/중/저) 매트릭스로 산정 |
| 5 | 위험성 | 자동: 유해성 × 노출수준 (1–16), §5.2.1 5단계 등급 판정 |
| 6 | 현재 관리 조치 | 국소배기, 밀폐, 보호구 등 텍스트 |
| 7 | 감소대책(5점 이상 필수) / 개선 후 등급 / 담당·기한 | 공통 규칙 동일 |

- FR-ASMT-C-001 (P0): 화학물질 마스터(물질명, CAS, 노출기준, GHS 분류, MSDS 파일)를 사업장별로 등록·관리할 수 있다.
- FR-ASMT-C-002 (P1): 유해성·노출수준 등급의 자동 제안 값을 평가자가 근거 입력과 함께 수동 조정할 수 있다(조정 이력 기록).
- FR-ASMT-C-003 (P2): 작업환경측정 결과(측정치, 측정일)를 물질·공정별로 입력·이력 관리하고 노출수준 산정에 재사용한다.

### 5.6 평가 일정·트리거 관리 (SCHD)

> 고시 제15조: 최초(성립 후 1개월 내 착수), 수시(사유 발생 시 작업 개시 전), 정기(매 1년), 상시(월·주·일 — 이행 시 수시+정기 갈음).

| ID | 우선순위 | 요구사항 |
|---|---|---|
| FR-SCHD-001 | P0 | 사업장별 평가 운영 모드를 선택한다: **「정기+수시」 모드** 또는 **「상시평가」 모드**. |
| FR-SCHD-002 | P0 | **정기평가 일정**: 사업장·부서별 최초/직전 정기평가 완료일 기준으로 차기 도래일(1년)을 자동 산출하고, D-60/D-30/D-7/기한 초과 시점에 담당자·안전관리자에게 알림을 발송한다. 기한 초과 건은 대시보드에 경보로 표시된다. |
| FR-SCHD-003 | P0 | **수시평가 트리거 이벤트 등록**: 사유 6종을 선택해 이벤트를 등록한다 — ① 건설물 설치·이전·변경·해체 ② 기계·기구·설비·원재료 신규 도입/변경 ③ 건설물·설비 정비·보수 ④ 작업방법·절차 신규 도입/변경 ⑤ 산업재해·아차사고 발생 ⑥ 중대산업사고 발생. 이벤트에는 대상 작업 개시 예정일을 입력한다. |
| FR-SCHD-004 | P0 | 트리거 이벤트에는 수시평가서가 연결되어야 하며, **작업 개시 예정일까지 승인 완료되지 않으면** 경보를 발생시킨다(수시평가는 작업 개시 전 실시 의무). 재해 사유(⑤⑥)는 재발방지대책 수립 전 실시 원칙을 안내 문구로 표시한다. |
| FR-SCHD-005 | P0 | **상시평가 모드 — 월 단위**: 월별로 '월 위험성평가(순회점검·근로자 제안·아차사고 확인 결과 반영)'를 1건 이상 완료해야 한다. 미완료 월은 경보 표시. 근로자 제보(FR-SHRE-005)와 아차사고 이벤트를 월 평가서에 연결할 수 있다. |

## 8. 데이터 모델 개요 (핵심 엔티티)

상세 스키마는 설계 단계 산출물로 하고, 여기서는 엔티티와 관계만 정의한다.

```
Company ─┬─ Site(사업장/현장) ─┬─ Dept(부서/공정, 트리)
         │                    ├─ Job(작업 마스터)
         │                    ├─ Chemical(화학물질 마스터, MSDS)
         │                    └─ PartnerContract(협력사 투입계약)
         ├─ Partner(협력사) ── PartnerContract
         └─ User(사용자) ── RoleAssignment(역할×스코프)

Assessment(평가서: 공통 헤더, 기법, 실시유형, 상태, 기준버전 스냅숏)
 ├─ AssessmentMember(평가팀·참여 근로자)
 ├─ HazardItem(요인 행: 공통 필드 + 기법별 확장 필드)
 │    ├─ (JSA) JobStep 소속        ├─ (HAZOP) Node·Deviation 소속
 │    └─ Action(개선조치 과제: 담당, 기한, 상태, 증빙, 재평가 결과)
 ├─ ApprovalFlow(결재선) ── ApprovalStep(단계별 처리 이력)
 ├─ SharePublication(공유 발행) ── Acknowledgement(열람 서명)
 └─ Attachment(첨부: 사진, MSDS, 도면)

RiskMatrixConfig(기법별 매트릭스·허용기준, 버전 관리)
ScheduleRule(정기 도래일) / TriggerEvent(수시 사유) / OngoingCycle(상시 월·주 이행 기록)
HazardReport(위험요인 제보) / Notification(알림) / AuditLog(감사 로그)
```

**모델링 원칙**

- `HazardItem`은 공통 컬럼 + 기법별 확장(JSON 또는 확장 테이블)으로 설계해 공통 엔진(결재·조치·공유·통계)이 기법과 무관하게 동작하게 한다.
- 승인 시점에 평가서 전체(적용 매트릭스 버전 포함)를 **스냅숏 보존**하여 이후 설정 변경이 과거 문서에 영향을 주지 않게 한다.
- 모든 도메인 테이블은 사업장(Site) 스코프 키를 가지며, 협력사 데이터는 Partner 키로 추가 격리한다.

### 8.1 Phase 1 MVP 구현 스키마·API·폴더 구조 (React + Supabase, v1.3)

> **v1.3 변경**: 본 절은 원래 `기획안.html`(vanilla JS+Express+SQLite 3일 프로토타입)과의 정합성 부록이었으나, 학습 목표를 제약에서 제외하고 실제 구현 스택을 **React(Next.js) + Supabase**로 확정함에 따라 전면 재작성한다. §8 본문의 엔티티(`Assessment`, `HazardItem` 등)는 전체 시스템(Phase 2~3)의 논리 모델이며, 아래는 KRAS 단일 기법으로 축소한 **Phase 1 구현용 Supabase(Postgres) 물리 스키마**다.

**테이블 매핑** (§8 엔티티 → Supabase 테이블)

| §8 엔티티 | Supabase 테이블 | 비고 |
|---|---|---|
| User | `profiles` (Supabase Auth `auth.users`를 1:1 확장) | 로그인·비밀번호 정책은 Supabase Auth가 처리(FR-ORG-008 대체) |
| Assessment | `assessments` | 실시유형·기법 선택·기준버전 스냅숏 등 엔터프라이즈 전용 필드(§5.3 공통 헤더)는 생략 — MVP는 KRAS 기법 고정 |
| HazardItem | `assessment_items` | §5.4.1 KRAS 13개 열 중 자동계산·필수검증에 관여하는 핵심 열만 반영 |
| Attachment(사진) | Supabase Storage 버킷 `assessment-photos` | FR-ASMT-008(요인 행 사진), FR-ACT-003(조치 증빙 사진) |

**Postgres DDL 개요**

```sql
create table departments (
  id bigint generated always as identity primary key,
  name text not null
);

create table profiles (
  id uuid primary key references auth.users(id),
  name text not null,
  role text not null check (role in ('작성자', '승인자')),
  department_id bigint references departments(id) -- §8.2 부서 기반 알림·자료 열람에 사용
);

create table assessments (
  id bigint generated always as identity primary key,
  title text not null,
  site text,
  department_id bigint references departments(id), -- 기존 dept(텍스트)를 FK로 대체 — §8.2
  previous_version_id bigint references assessments(id), -- §8.2 개정(보완) 연결
  evaluator_id uuid references profiles(id),
  status text not null default '작성중'
    check (status in ('작성중', '결재대기', '승인', '반려')),
  created_at timestamptz default now()
);

create table assessment_items (
  id bigint generated always as identity primary key,
  assessment_id bigint references assessments(id) on delete cascade,
  hazard text not null,
  current_action text,
  frequency int not null check (frequency between 1 and 4),
  severity int not null check (severity between 1 and 4),
  risk_score int generated always as (frequency * severity) stored, -- §5.2.1 4×4 매트릭스
  mitigation text,
  photo_path text, -- Storage 버킷 내 경로
  created_at timestamptz default now(),
  constraint mitigation_required_over_threshold
    check (risk_score < 5 or mitigation is not null) -- FR-ASMT-003
);
```

- `risk_score`는 Postgres **generated column**으로 DB가 자동 계산한다(§5.2.1 매트릭스, FR-ASMT-003).
- `mitigation_required_over_threshold` CHECK 제약으로 "위험성 5점 이상은 감소대책 필수" 규칙(FR-ASMT-003)을 **DB 레벨에서 강제**한다 — 클라이언트(React) 검증이 우회되어도 서버(DB)가 최종 방어선이 된다.
- 세부 사업장·역할별 접근 제어(NFR-006)는 Supabase **Row Level Security(RLS)** 정책으로 구현한다(예: "본인이 작성한 평가서만 수정 가능", "승인자 역할만 status를 '승인'/'반려'로 변경 가능"). MVP 단계는 단순 정책(로그인 사용자 전체 조회·본인 글만 수정)으로 시작하고, Phase 2에서 조직 스코프 정책으로 확장한다.

**로그인·사진·알림 구현 방식**

| 기능 | 구현 |
|---|---|
| 로그인(FR-ORG-008) | Supabase Auth(`supabase.auth.signInWithPassword` 등). 자체 비밀번호 해싱·잠금 로직 불필요 |
| 사진 등록(FR-ASMT-008, FR-ACT-003) | Supabase Storage 버킷 `assessment-photos`에 업로드 후 반환된 경로를 `assessment_items.photo_path`(또는 조치 테이블)에 저장. 로컬 디스크 저장이 아니므로 배포 후 재시작해도 파일이 유지된다 |
| 알림(FR-NOTI-001) | MVP는 `notifications` 테이블 + 단순 폴링(`GET /api/notifications`)으로 충분. Supabase Realtime 구독은 실시간성이 필요해지는 Phase 2 이후 검토 |

**관리자 계정 로그인 테스트 절차 (초기 개발 단계)**

회원가입 화면을 만들기 전에 관리자 계정 로그인만 먼저 검증하는 순서:

1. Supabase 대시보드 → Authentication → Users → "Add user"로 관리자 계정을 직접 생성(이메일/비밀번호 지정). Settings에서 이메일 인증 확인 절차를 꺼두면 가입 즉시 로그인 가능.
2. `profiles` 테이블에 해당 계정과 연결된 행을 추가하고 `role = '승인자'`(관리자)로 지정.
3. 앱에서 `supabase.auth.signInWithPassword({ email, password })`로 로그인 처리 — 회원가입 폼·비밀번호 해싱·세션 관리 코드를 직접 구현할 필요 없음.
4. 로그인 성공 후 `profiles.role`을 조회해 관리자 화면으로 분기.

> **주의**: 이 단계에서 테이블에 RLS(행 단위 접근 제한)를 걸어두면 정책을 잘못 설정할 경우 관리자 계정 본인도 데이터를 못 읽을 수 있다. 로그인 흐름만 검증하는 동안은 RLS를 끄거나 "로그인한 사용자는 전체 허용" 수준의 느슨한 정책으로 시작하고, 흐름이 확인되면 위 모델링 원칙(§8 본문)과 NFR-006에 맞춰 세밀한 정책으로 조인다.

### 8.2 추가 기능 6종 — 스키마·로직 매핑 (2026-07-08 논의 반영)

> 이 절은 "데이터 누적 후 대시보드 가시화 / 고위험 부서 알림 / 타 부서 자료 열람 / 기존 자료 보완 / 입력 가이드라인 / 법령·재해사례 참고자료"의 6개 기능을 §8.1 스키마에 반영한 것이다. 6개 모두 **Express 등 별도 백엔드 서버 없이** Postgres 함수·트리거·RLS·전문검색과 Next.js API Route로 처리한다. 아래 기능 중 정식 FR로 확정되지 않은 것은 §5 요구사항 목록에 추가 여부를 별도 결정한다.

**2) 고위험 자동 알림 — Postgres 트리거**

```sql
create table notifications (
  id bigint generated always as identity primary key,
  user_id uuid references profiles(id),
  message text not null,
  assessment_item_id bigint references assessment_items(id),
  read_at timestamptz,
  created_at timestamptz default now()
);

create function notify_high_risk() returns trigger as $$
begin
  if new.risk_score >= 5 then
    insert into notifications (user_id, message, assessment_item_id)
    select p.id,
           format('[고위험 알림] 위험성 %s점 요인이 등록되었습니다.', new.risk_score),
           new.id
    from profiles p
    join assessments a on a.id = new.assessment_id
    where p.department_id = a.department_id;
  end if;
  return new;
end;
$$ language plpgsql security definer;

create trigger trg_notify_high_risk
  after insert or update of frequency, severity on assessment_items
  for each row execute function notify_high_risk();
```

- 클라이언트가 알림 발송을 "잊어버릴" 여지가 없도록 저장 시점에 DB가 강제 실행한다(§8.1 CHECK 제약과 같은 설계 원칙).
- 알림함 화면(§7 S-03)은 `notifications` 테이블을 사용자별로 폴링 조회한다.

**3) 다른 부서 자료 열람 — RLS 정책**

```sql
create policy "read_all_assessments" on assessments
  for select using (true); -- 전 부서 열람 허용

create policy "update_own_department_only" on assessments
  for update using (
    department_id = (select department_id from profiles where id = auth.uid())
  );
```

**4) 이전 자료 보완 — Postgres 함수(개정)**

```sql
create function create_revision(p_assessment_id bigint) returns bigint as $$
declare
  new_id bigint;
begin
  insert into assessments (title, department_id, evaluator_id, status, previous_version_id)
  select title, department_id, evaluator_id, '작성중', id
  from assessments where id = p_assessment_id
  returning id into new_id;

  insert into assessment_items (assessment_id, hazard, current_action, frequency, severity, mitigation, photo_path)
  select new_id, hazard, current_action, frequency, severity, mitigation, photo_path
  from assessment_items where assessment_id = p_assessment_id;

  return new_id;
end;
$$ language plpgsql security definer;
```

- 기존 평가서는 그대로 보존(읽기전용), 새로 생성된 행이 `previous_version_id`로 원본을 참조한다(§5.3 FR-ASMT-010 개정 규칙과 동일한 원칙).

**5) 입력 가이드라인 — 클라이언트 + 기존 DB 제약**

별도 테이블 불필요. React 폼에서 실시간으로 경고를 띄우고(예: "위험성 5점 이상인데 감소대책 미입력"), 최종 저장은 §8.1의 `mitigation_required_over_threshold` CHECK 제약이 막는다. 향후 "과거 유사 요인 추천" 같은 고급 가이드로 확장하려면 6번의 `assessment_items` 텍스트 검색을 재사용할 수 있다.

**6) 법령·재해사례 참고자료 — 전문검색**

```sql
create table reference_materials (
  id bigint generated always as identity primary key,
  category text check (category in ('법령', '재해사례')),
  title text not null,
  content text not null,
  search_vector tsvector generated always as (to_tsvector('simple', title || ' ' || content)) stored,
  created_at timestamptz default now()
);

create index reference_materials_search_idx on reference_materials using gin(search_vector);
```

> 주의: `'simple'` 검색설정은 공백·구두점 기준 토큰화만 하며 한국어 형태소 분석은 하지 않는다(예: "위험성평가"와 "위험성평가를"을 다른 단어로 취급). 정확한 키워드 검색에는 충분하지만, 더 정교한 한국어 검색이 필요해지면 `pg_bigm` 확장 등을 Phase 2에서 검토한다.

### 8.3 JSA 데이터 모델 상세 (엔터프라이즈 전체모델 예시)

> 원격 저장소(GitHub `day10`)의 최신 `PRD.md`에 추가된 JSA 논리 데이터 모델을 반영한 절이다. 원본은 "§8.1"로 번호가 매겨져 있었으나 본 문서의 §8.1(Phase 1 Supabase 물리 스키마)과 번호가 충돌하므로 §8.3으로 재배치했다. 아래는 §8 본문 엔티티의 JSA 확장을 다룬 **전체 시스템(Phase 2~3) 논리 모델**이며, §8.1의 Supabase MVP 스키마와는 별개다 — MVP가 JSA까지 확장되는 시점에 상호 대조해 반영한다.

```
Assessment (기법='JSA', 실시유형='수시')
 └─ AssessmentJSA (1:1 확장)
      - 작업위치 (VARCHAR)
      - 작업예정일시 (DATETIME)
      - 필요자격면허 (VARCHAR[] / JSON)
      - 필요보호구체크리스트 (JSON: [{item, required}])
      - 관련작업허가참조 (VARCHAR, nullable)
      - trigger_event_id (FK → TriggerEvent, nullable — 수시 트리거로 생성된 경우)

Assessment(JSA) 1 ── N JobStep
 JobStep
  - id (PK)
  - assessment_id (FK)
  - seq_no (INT, 정렬 순번 — assessment_id 범위 내 유일)
  - step_name (VARCHAR, 동사형 서술)
  - step_desc (TEXT)
  - equipment_tools (VARCHAR[] / JSON)

JobStep 1 ── N HazardItem (JSA 확장)
 HazardItem
  - id (PK)
  - job_step_id (FK)
  - hazard_type_code (FK → 유해·위험요인 분류코드 마스터)
  - hazard_desc (TEXT)
  - injury_type_code (FK → KOSHA 재해분류 마스터)
  - current_measure (TEXT)
  - freq (INT 1–4) / severity (INT 1–4)
  - risk_score (INT, computed = freq × severity)
  - risk_grade (ENUM: 매우위험/위험/다소위험/위험성존재/허용가능, computed)
  - needs_action (BOOLEAN, computed = risk_score ≥ 5)
  - reduction_measure_type (ENUM: 제거대체/공학적/관리적/개인보호구, needs_action=true면 필수)
  - reduction_measure_desc (TEXT, needs_action=true면 필수)
  - residual_freq / residual_severity / residual_risk_score (개선 후 위험성)
  - matrix_version_id (FK → RiskMatrixConfig, 작성 시점 스냅숏)

HazardItem 1 ── 0..1 Action (needs_action=true면 필수 생성)
 Action
  - id (PK)
  - hazard_item_id (FK)
  - owner_user_id (FK → User)
  - due_date (DATE, 등급별 디폴트 자동제안 + 수정가능)
  - status (ENUM: 대기/진행중/완료/지연)
  - evidence_attachment_id (FK → Attachment, nullable)
  - verified_by / verified_at (완료 확인자)

HazardItem 1 ── N Attachment (현장 사진 등)
```

**JSA 고유 제약조건**

- `AssessmentJSA.trigger_event_id`가 존재하면, 해당 Assessment는 `TriggerEvent.target_start_date`(작업 개시 예정일)까지 상태가 '승인완료'로 전이되지 않는 한 배치 잡이 매일 경보(Notification)를 생성한다(FR-SCHD-004).
- `HazardItem.risk_score`가 5 이상인데 `reduction_measure_type`이 NULL이면, Assessment 상태를 '작성중'→'결재중'으로 전이하는 API가 검증 오류를 반환한다(FR-ASMT-003의 서버 측 강제 로직).
- `JobStep.seq_no` 재정렬(드래그)은 assessment_id 범위 내 일괄 갱신(bulk update) 트랜잭션으로 처리해 순번 충돌을 방지한다.

## 10. 릴리스 단계 계획

| Phase | 범위 | 포함 요구사항 |
|---|---|---|
| **Phase 1 — 코어 (MVP)** | KRAS·JSA 작성 + 결재 + 조치 이행 + 공유·서명 + 기록 | ORG 전체(P0), CFG-001~006, ASMT 공통(P0)·K·J, APRV-001~005, ACT-001~006, SHRE-001~004, REC-001~004, SCHD-002~004(정기+수시), NOTI-001, DASH-002 |
| **Phase 2 — 기법 확장 + 상시평가** | HAZOP·CHARM, 상시평가 모드, 협력사 포털 보강, 제보 | ASMT-H·C, SCHD-001/005~009, SHRE-005~006, APRV-006~007, ACT-007~008, DASH-001/003/005, REC-005 |
| **Phase 3 — 관제 고도화** | 통계·리포트, 요인 라이브러리 고도화, 운영 편의 | DASH-004/006, ASMT-006/007/010, SHRE-007, ORG-009, APRV-008, ACT-009, CFG-007, REC-006, NOTI-003 |

- Phase 1의 완료 기준: 1개 파일럿 사업장에서 정기(KRAS)·협력사(JSA) 평가의 작성→결재→공유·서명→조치 완료 사이클이 시스템만으로 완결되고, 증빙 패키지가 출력된다.
- 상시평가 모드(Phase 2)는 파일럿 사업장의 정기+수시 운영이 안정된 뒤 전환한다.
- **3일 MVP 프로토타입(`기획안.html`)과 Phase 1의 관계**: 3일 프로토타입은 Phase 1 착수를 위한 선행 스프린트로, KRAS 단일 기법·단일 사업장·간이 로그인 가정 하에 핵심 흐름(작성 → 회사 표준 4×4 매트릭스 자동계산 → 결재 → 대시보드 집계)만 검증한다(협력사·JSA·3단계 결재선·공유서명·조직관리는 제외, §8.1). 즉 **3일 프로토타입 ⊂ Phase 1**이며, 위 Phase 1 완료 기준을 충족하려면 협력사(JSA) 연동, 정식 인증(FR-ORG-008), 조직·사업장 관리(FR-ORG-001~006)까지 확장이 필요하다.

## 12. 용어집

| 용어 | 정의 |
|---|---|
| 위험성평가 | 유해·위험요인을 파악하고 부상·질병의 가능성과 중대성을 고려해 위험성 수준을 결정하고 감소대책을 수립·실행하는 일련의 과정 (산안법 제36조) |
| HAZOP | Hazard and Operability Study. 공정 변수×가이드워드로 설계 의도로부터의 이탈을 분석하는 공정 위험성평가 기법. 본 시스템에서는 **산업안전보건법상 공정안전자료(PSM)**에 포함되는 공정위험성평가 양식으로 사용 |
| KRAS | 안전보건공단 위험성평가 지원시스템에서 유래한 빈도×강도 기반 평가 방식. 본 시스템에서는 **정기 사업장 위험성평가**에 쓰는 빈도·강도법 시트를 지칭 |
| JSA | Job Safety Analysis. 작업을 단계로 분해하여 단계별 유해·위험요인과 대책을 분석하는 작업 위험성평가 기법. 본 시스템에서는 **신규 작업 발생 시 작업 착수 전** 실시하는 수시평가 양식으로 사용 |
| CHARM | Chemical Hazard Risk Management. 화학물질의 유해성 등급×노출수준 등급으로 위험성을 산정하는 KOSHA 화학물질 평가 기법. 본 시스템에서는 **저장·취급 화학물질**에 대한 물질위험성평가 양식으로 사용 |
| 회사 표준 매트릭스 | 4개 기법에 공통 적용하는 4×4(1~16점) 위험성 결정 매트릭스. 5단계 등급(매우위험 13~16/위험 10~12/다소위험 7~9/위험성존재 5~6/허용가능 1~4)과 등급별 조치기한을 정의 (§5.2.1) |
| 상시평가 | 월(요인 파악·평가)·주(공유·이행점검)·일(작업 전 주지) 주기로 실시하여 수시·정기평가를 갈음하는 제도 (고시 제15조) |
| 수시평가 트리거 | 수시평가 실시 사유 6종(설비 도입·변경, 정비·보수, 작업방법 변경, 재해 발생 등) |
| 감소대책 우선순위 | 제거·대체 > 공학적 대책 > 관리적 대책 > 개인보호구 순의 법정 우선순위 (고시 제12조) |
| 아차사고 | 부상·질병으로 이어지지 않았으나 그럴 가능성이 있었던 사건. 수시평가 사유이자 상시평가 파악 대상 |
| 도급인/수급인 | 자사(도급인)와 협력사(수급인). 수급인은 자기 작업의 평가를 실시하고 도급인은 이를 확인할 의무가 있다 |
