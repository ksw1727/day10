# LEARNING_GUIDE.html 설계안

- 문서 버전: v1.0
- 작성일: 2026-07-08
- 관련 문서: `PRD.md`, `기획안.html` (§8 필요한 학습 개념)

## 1. 배경 및 목적

`기획안.html` §8 "필요한 학습 개념"은 팀원별로 배워야 할 개념명만 나열되어 있다(예: "DOM 선택·생성·삭제", "SQL 기초: CREATE TABLE, SELECT, INSERT, UPDATE, 외래키"). 이 목록만으로는 팀원이 "왜 이걸 배워야 하는지"와 "이 프로젝트의 어느 부분에 실제로 쓰이는지"를 알기 어렵다.

이 설계는 §8을 팀원이 학습하면서 바로 옆에 펴놓고 참조할 수 있는 **실전 학습 가이드**로 확장한다. 일반적인 튜토리얼이 아니라, 3일 MVP 기획안에 정의된 실제 폴더 구조·API 명세·DB 스키마·KRAS 13개 열 정의를 그대로 예시에 사용해 "이 프로젝트에서 이 개념이 정확히 이렇게 쓰인다"를 보여준다.

## 2. 범위

**포함**

- 새 파일 `LEARNING_GUIDE.html` 1개 생성 (프로젝트 루트, `기획안.html`과 같은 위치)
- `기획안.html`의 CSS(색상 변수, 카드·표 스타일)를 그대로 재사용해 시각적으로 통일된 문서 작성
- 기획안 §8에 이미 나열된 개념을 기준으로, 각 개념에 "왜 / 어디서 / 예시" 3단 구조를 채워 넣음
- 예시 코드는 기획안 §6(API 목록), §7(폴더 구조), PRD §5.4.1(KRAS 13개 열)에 정의된 실제 이름·경로·스키마를 그대로 사용

**제외**

- 기획안.html §8 자체의 수정/삭제 (그대로 유지, 새 문서로 분리하기로 확정됨)
- 실제 코드 구현 (이 문서는 학습 자료이며 동작하는 코드가 아님 — 예시는 발췌 스니펫)
- 팀원 A/B/C 외 새로운 역할 추가, 개념 항목 자체의 확대(예: 새로운 기술 스택 추가) — 기존 §8에 나온 개념만 구체화 대상

## 3. 문서 구조

파일: `LEARNING_GUIDE.html`

```
공통 (전원, Day 1 전까지)
 ├─ Git 기본 (clone/branch/add/commit/push/pull, PR·머지)
 ├─ JSON 구조 읽고 쓰기
 └─ 도메인 지식: 회사 표준 4×4 매트릭스, 5단계 등급·조치기한, 감소대책 우선순위

팀원 A — 화면
 ├─ CSS Flexbox/Grid
 ├─ <table> 마크업과 폼 요소(select, textarea)
 └─ 상태·등급 색상 클래스 표현

팀원 B — 프론트 로직
 ├─ DOM 선택·생성·삭제
 ├─ 이벤트 처리(addEventListener, 이벤트 위임)
 ├─ 배열 메서드(map/filter/reduce)
 └─ fetch + async/await

팀원 C — 백엔드
 ├─ npm, Express 라우팅과 미들웨어
 ├─ REST API 설계 기초
 ├─ SQL 기초(CREATE/SELECT/INSERT/UPDATE, 외래키)
 └─ better-sqlite3 사용법

여유가 있으면 (선택)
 ├─ 트랜잭션
 ├─ 서버 측 입력값 검증
 └─ 배포(Render/Railway 등)
```

이 구조는 기획안.html §8과 1:1로 대응한다(항목 추가/삭제 없음, 각 항목의 내용만 3단 구조로 확장).

## 4. 개념별 콘텐츠 템플릿

각 개념은 다음 3개 구성요소로 작성한다.

```
### [개념명] (필수 / 선택 배지)
**왜**: 이 프로젝트의 어떤 기능·요구사항 때문에 필요한지. 가능한 경우 PRD 조항(§번호) 또는
        기획안 섹션을 인용한다.
**어디서**: 기획안 §7 폴더 구조에 정의된 실제 파일 경로 중 이 개념이 쓰이는 곳
           (예: server/db.js, public/js/form.js, routes/assessments.js)
**예시**: 이 프로젝트의 실제 데이터(예: assessments/assessment_items 테이블,
         KRAS 13개 열, /api/assessments 엔드포인트)를 사용한 짧은 코드 스니펫
```

예시 (팀원 C, SQL 외래키):

```
### SQL 외래키 (필수)
**왜**: 평가서 1건에 요인 행이 여러 개 달리는 1:N 구조를 표현해야 함 (PRD §5.3)
**어디서**: server/db.js의 assessment_items 테이블이 assessment_id를 FK로 가짐
**예시**:
CREATE TABLE assessment_items (
  id INTEGER PRIMARY KEY,
  assessment_id INTEGER REFERENCES assessments(id),
  frequency INTEGER, severity INTEGER
);
```

모든 예시는 실제 동작 검증 대상이 아닌 학습용 발췌 코드이며, 실제 구현 시 세부 컬럼·검증 로직은 달라질 수 있다.

## 5. 시각 디자인

`기획안.html`과 통일감을 위해 동일한 CSS를 재사용한다.

- 동일 CSS 변수(`--ink`, `--sub`, `--line`, `--accent`, 등급 색상 `.g1~.g5`)
- 동일 카드(`section.card`), 배지(`.pill`, `.lv`), 표(`table`) 스타일
- 개념별 블록은 `기획안.html`의 `.learn` 카드 스타일을 확장해 "왜/어디서/예시" 3단을 하위 구조로 표시(예: 라벨 + 텍스트, 예시는 `<pre><code>` 블록)
- 필수/선택 배지는 기존 `.lv` 클래스 재사용(필수=강조색, 선택=회색)
