위험성평가 시스템 — 3일 MVP 기획안

3인 팀 · 백엔드 포함 풀스택 · 상세 요구사항은 PRD.md 참조 (본 기획안은 PRD Phase 1에서 3일 분량을 다시 잘라낸 것)

1문제

우리 팀이 풀려는 문제 한 문장.

건설·제조 사업장의 위험성평가가 엑셀과 수기 문서에 흩어져 있어, 작성 → 결재 → 개선조치 → 기록의 흐름이 추적되지 않고 법정 의무(정기평가 연 1회, 기록 3년 보존, 근로자 공유)를 놓치기 쉽다. 근거: 산업안전보건법 제36조, 시행규칙 제37조, 고용노동부 고시 제2024-76호 — 자세한 내용은 PRD §1, §9

2개발 목표 (MVP)

3일 안에 만들 첫 버전의 범위.

한 줄 목표

KRAS(빈도·강도법) 평가서 하나를 "작성 → 자동 위험도 계산 → 목록·상세 → 승인/반려 → 대시보드 집계"까지 웹에서 끝내는 것.

포함 / 제외

구분내용
포함
  • 간단 로그인 (이름 선택 방식 — 작성자/승인자 구분용, 회원가입 없음)
  • KRAS 평가서 작성: 공정/작업명, 유해·위험요인, 현재 조치, 빈도(1–3) × 강도(1–3) 선택 → 위험성 자동 계산·등급 색상 1–2 낮음 3–4 보통 6 높음 9 매우높음
  • 위험성 3 이상 행에는 감소대책 입력 필수 (미입력 시 제출 차단)
  • 평가서 목록 (상태 필터: 작성중 / 결재대기 / 승인 / 반려)
  • 평가서 상세 + 승인/반려 (반려 사유 필수)
  • 대시보드: 전체 건수, 상태별 건수, 고위험(6 이상) 항목 수
  • 데이터는 서버 DB(SQLite)에 저장 — 새로고침해도 유지
제외 HAZOP·JSA·CHARM 기법, 협력사 제출, 3단계 결재선, 열람 전자서명, 개선조치 이행관리(증빙사진), 정기/수시/상시 일정 관리, 알림, 권한 관리 — 전부 PRD에 정의돼 있고 MVP 이후 확장

일자별 계획

Day 1 — 뼈대

  • 폴더·Git 저장소 세팅, API 명세 합의
  • DB 테이블 설계(평가서, 요인 행)
  • 화면 4개 HTML/CSS 뼈대
  • 서버 실행 + 목록 조회 API 1개 연결

Day 2 — 핵심 기능

  • 평가서 작성 폼: 행 추가/삭제, 빈도×강도 자동 계산
  • 저장(등록/수정) API, 목록·상세 연결
  • 감소대책 필수 검증

Day 3 — 결재·대시보드·마무리

  • 제출 → 승인/반려 처리
  • 대시보드 집계 API + 화면
  • 버그 수정, 데모 시나리오 리허설

최종 로드맵 (MVP 이후)

KRAS 구조를 공통 엔진으로 삼아 JSA → CHARM → HAZOP 순으로 기법을 확장하고, 이후 결재선·공유서명·이행관리·일정관리를 PRD Phase 1~3 순서로 붙인다.

3화면

필요한 화면과 화면 간 이동.

S1 로그인이름 선택 + 역할(작성자/승인자)
→
S2 대시보드건수 카드 3개 + 고위험 수
→
S3 평가서 목록상태 필터, 새 평가서 버튼
→
S4 평가서 작성/수정헤더 + 요인 행 표, 자동 계산
→
S5 평가서 상세읽기 전용 + 승인/반려 버튼
화면주요 요소이동
S1 로그인사용자 선택 드롭다운(시드 데이터 3명), 시작 버튼→ S2
S2 대시보드전체/결재대기/승인 건수 카드, 고위험 항목 수, 최근 평가서 5건카드 클릭 → S3(필터 적용)
S3 목록표(번호·제목·부서·상태·고위험 수·작성일), 상태 필터 탭, [새 평가서]행 클릭 → S5, 새 평가서 → S4
S4 작성/수정헤더(제목·사업장·부서·평가자), 요인 행 표(요인·현재조치·빈도·강도·위험성·감소대책), [행 추가] [임시저장] [제출]저장/제출 → S5
S5 상세전체 내용 읽기 전용, 상태 배지, (승인자일 때) [승인] [반려+사유], (반려 시 작성자) [수정]수정 → S4, 처리 후 → S3

공통 레이아웃: 상단 바(로고 · 대시보드 · 평가서 목록 · 사용자명). 전체 화면은 PC 우선, 모바일은 열람만 가능한 수준으로 대응.

4역할

누가 무엇을 맡는지. (3인 — 이름은 팀에서 채워 넣기)

👤 팀원 B — 프론트 로직 담당

산출물: server/, API 명세 문서

5협업 방식

팀을 어떻게 나눠 만들지.

  • 계약 먼저, 코드 나중: Day 1 오전에 ① 화면 5개의 요소 ② API 명세(주소·요청·응답 JSON 예시) ③ DB 컬럼을 종이 한 장으로 합의하고 시작한다. 이후 프론트(A·B)와 백엔드(C)는 명세만 보고 병렬 작업.
  • 백엔드가 늦어도 멈추지 않기: B는 API가 나오기 전까지 가짜 데이터(mock.js)로 화면을 완성하고, API가 준비되면 주소만 바꿔 연결한다.
  • Git 규칙: main은 항상 실행 가능한 상태 유지 · 작업은 feat/화면이름 브랜치 → Pull Request → 다른 팀원 1명 확인 후 머지 · 커밋 메시지는 feat: 평가서 목록 화면 형식.
  • 하루 두 번 동기화: 아침 10분(오늘 할 일·막힌 것 공유) + 저녁 머지 타임(그날 브랜치 전부 main에 합치고 함께 실행 확인).
  • 막히면 30분 규칙: 30분 이상 혼자 막히면 바로 팀에 공유한다.

6사용 기술

어떤 기술과 라이브러리로 만들지.

영역기술선택 이유
프론트엔드HTML + CSS + 바닐라 JavaScript팀 수준(JS 입문)에 맞고, 프레임워크 학습 없이 3일 안에 완성 가능
서버 통신fetch API + JSON브라우저 내장 — 별도 라이브러리 불필요
백엔드Node.js + Express프론트와 같은 언어(JS)라 학습 부담 최소, 라우팅이 단순
데이터베이스SQLite (better-sqlite3)설치·서버 구동 없이 파일 하나로 동작, 이후 기법 확장 시 테이블만 추가
협업Git + GitHub브랜치·PR 협업, 형상 관리
개발 도구VS Code, 크롬 개발자도구, Postman(또는 Thunder Client)API를 화면 없이 먼저 테스트

8필요한 학습 개념

이 과제를 만들려면 무엇을 배워야 하는지. (현재 수준: HTML/CSS 기초 · JS 입문 → 역할별로 필요한 것만 학습)

공통 (전원, Day 1 전까지)필수

  • Git 기본: clone / branch / add / commit / push / pull, PR 만들기·머지
  • JSON 구조 읽고 쓰기 (객체·배열 중첩)
  • 도메인 지식: 위험성평가 5단계 절차, KRAS 빈도(1–3)×강도(1–3) 계산과 등급 기준, 감소대책 우선순위(제거·대체>공학>관리>보호구) — PRD §5.3~5.4 읽기

팀원 A — 화면필수

  • CSS Flexbox/Grid로 카드·표 레이아웃
  • 표(<table>) 마크업과 폼 요소(select, textarea)
  • CSS 클래스로 상태·등급 색상 표현

팀원 B — 프론트 로직필수

  • DOM 선택·생성·삭제 (querySelector, createElement)
  • 이벤트 처리 (addEventListener, 이벤트 위임)
  • 배열 메서드 (map, filter, reduce)로 계산·렌더링
  • fetch + async/await로 API 호출, 에러 처리

팀원 C — 백엔드필수

  • npm 사용법, Express 라우팅 (GET/POST/PUT)과 미들웨어(express.json, 정적 파일)
  • REST API 설계 기초 (경로·메서드·상태코드)
  • SQL 기초: CREATE TABLE, SELECT, INSERT, UPDATE, 외래키(평가서–요인 행 1:N)
  • SQLite를 better-sqlite3로 다루기

여유가 있으면선택

  • 트랜잭션 (평가서+행 여러 개를 한 번에 저장)
  • 입력값 서버 측 검증
  • 배포 (Render, Railway 등 무료 호스팅)
